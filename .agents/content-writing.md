# 교육 콘텐츠 집필 지침

이 문서는 `docs/phase-*`와 `docs/appendix-*`의 챕터를 새로 쓰거나 수정·검토할 때 적용한다. `plan/`의 기획 문서와 `exercises/`의 실습 문서에는 이 구조를 강제하지 않는다. 주제 범위나 문서 구성을 바꾸는 작업은 [커리큘럼 관리 지침](curriculum-management.md)도 함께 따른다.

## 독자와 깊이

독자는 프론트엔드가 주력이 아니었던 5년차 이상 경력 개발자다. 프로그래밍 일반이나 Git·HTTP의 존재 이유 같은 기초를 반복하지 말고, 웹 플랫폼 고유의 동작 원리와 설계 판단에 집중한다. 독자의 서버·모바일·타 언어 경험과 연결되는 비교는 멘탈 모델을 세우는 데 필요한 경우 활용한다.

모든 주요 주제는 다음 세 층위를 갖춘다.

1. **동작 모델** — 표면 API 아래에서 파서·엔진·런타임·네트워크가 어떻게 동작하는지 설명한다.
2. **설계 배경** — 어떤 문제와 대안이 있었고 현재 설계가 무엇을 얻고 포기했는지 설명한다.
3. **경계 조건** — 추상화가 무너지는 상황, 성능이 급락하는 지점, 다른 선택이 필요한 신호를 설명한다.

사용법만 나열한 주요 섹션은 완성된 것으로 보지 않는다.

## 기본 문서 구조

챕터와 부록은 다음 흐름을 기본으로 삼는다.

1. `# <챕터 번호>. <제목>`과 판단·구현 결과를 나타내는 한 줄 요약
2. `## 학습 목표` — 설명·판단·구현처럼 검증 가능한 동사로 작성한 3~5개 목표
3. `## 배경: 왜 이것이 존재하는가` — 해결하려는 문제, 등장 배경, 대안
4. `## 핵심 개념` — 동작 원리를 중심으로 구성한 본문
5. `## 실무 관점` — 오해, 안티패턴, 대안별 트레이드오프와 경계 조건
6. `## 더 깊이` — 스펙이나 구현 세부. 본문에서 이미 충분히 다뤘다면 생략할 수 있다.
7. `## 정리` — 새 정보를 넣지 않은 핵심 요약
8. `## 확인 문제` — 원인 분석이나 설계 선택을 요구하는 2~4개 문제와 접힌 정답·해설
9. `## 참고 자료` — 1차 자료를 우선한 출처 목록과 각 자료의 용도

이 순서는 일관된 학습 흐름을 위한 기본값이다. 부록의 회고 질문, 사례 분석, ADR처럼 주제에 필요한 섹션은 추가할 수 있다.

## 서술 방식

- 한국어 평서형(`~한다`, `~이다`)으로 작성하고 동료 시니어에게 설명하는 담백한 어조를 유지한다.
- 기술 용어는 첫 등장에 원어를 병기한다. 번역이 어색하거나 원어가 통용되면 이후에는 원어를 사용한다.
- 원리와 제약을 먼저 세우고 API를 그 모델의 표현으로 설명한다.
- 스펙의 용어와 모델을 기준으로 삼는다. 통념이 다르면 차이가 생긴 이유까지 설명한다.
- 표준이 보장하는 동작과 특정 엔진·브라우저·프레임워크의 구현 세부를 구분한다.
- 성능이나 비용 주장은 DevTools 관찰 절차, 재현 방법, 벤치마크 등 확인 방법을 함께 제시한다.
- 역사적 맥락과 비유는 현재의 설계를 이해하는 데 필요한 만큼만 사용한다.
- 장점은 마케팅 표현이 아니라 근거와 비용으로 설명한다.

## 예제와 관찰

- 코드로 관찰할 수 있는 주제는 복사해 실행 가능한 최소 예제와 예상되는 관찰 결과를 제공한다.
- 기준 버전과 브라우저·런타임·프레임워크 등 실행 환경을 결과 해석에 필요한 수준으로 밝힌다.
- 코드 일부, 설정 조각, 의사 코드, 구조도는 완전한 실행 예제로 오해하지 않도록 성격과 생략 범위를 표시한다.
- 나쁜 접근을 소개하면 대안과 함께 비교하되 특정 아이콘이나 주석 형식을 강제하지 않는다.
- 식별자는 영어로 쓰고, 설명이 필요한 주석은 한국어로 작성한다.
- 내부 동작은 가능하면 실행 순서 출력이나 DevTools 절차처럼 독자가 직접 확인할 수 있게 한다.

## 정확성과 출처

- 확신이 없는 스펙, 버전, 수치, 호환성은 추측하지 않고 현재의 공식 자료를 확인한다.
- 스펙, RFC, 공식 문서와 공식 소스를 우선하고 블로그는 보조 검증 자료로만 사용한다.
- 특정 라이브러리나 프레임워크를 다루면 기준 버전을 문서에 명시한다.
- 브라우저 지원이 갈리는 기능은 현재 지원 상태와 판단 근거를 함께 제시한다.
- 구현 세부에 의존하는 코드를 권장하지 않는다.
- 참고 링크에는 독자가 해당 자료에서 무엇을 확인할 수 있는지 짧게 설명한다.

## 분량과 분리

챕터 하나를 읽고 예제를 따라 하는 데 45분~1시간 30분 정도를 참고 기준으로 삼는다. 분량이 커지면 깊이를 줄이지 말고 주제의 폭을 나눈다. 문서를 분리하거나 범위를 바꿀 때는 [커리큘럼 관리 지침](curriculum-management.md)에 따라 ROADMAP과 관련 자료를 함께 정리한다.

## 완료 전 점검

- 독자가 이미 아는 일반 프로그래밍 설명이 본론을 가리지 않는가?
- 주요 주제가 동작 모델·설계 배경·경계 조건을 갖췄는가?
- 선택지와 성능·비용 주장에 근거와 확인 방법이 있는가?
- 실행 예제와 관찰 결과를 실제 환경에서 검증했는가?
- 확인 문제가 암기가 아니라 원인 분석이나 판단을 요구하는가?
- 내부 링크와 참고 링크가 의도한 대상을 가리키는가?
