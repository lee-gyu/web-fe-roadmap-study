# CLAUDE.md — 교육 문서 집필 지침

이 저장소는 경력 개발자를 위한 웹 프론트엔드 **심화(deep dive) 교육 문서** 모음입니다.
전체 커리큘럼과 문서 배치 계획은 [ROADMAP.md](ROADMAP.md)를 따릅니다.
이 문서는 각 학습 챕터(교육 문서)를 작성할 때 반드시 지켜야 할 지침입니다.

## 1. 독자 정의 (가장 중요)

**독자는 5년차 이상의 경력 개발자입니다.** 프론트엔드가 주력이 아니었을 뿐(백엔드, 모바일 등), 프로그래밍 일반에는 이미 능숙합니다. 이 독자가 원하는 것은 튜토리얼이 아니라 **내부 동작 원리와 설계 판단 기준**입니다.

이 전제에서 다음을 지킵니다.

- **설명하지 않는 것**: 변수/조건문/반복문 같은 프로그래밍 기초, 함수형/객체지향의 일반 개념, Git 사용법 일반, HTTP의 존재 이유 같은 상식 수준의 내용. 이런 내용이 필요하면 한 문장으로 언급하고 지나간다.
- **집중해서 설명하는 것**: 프론트엔드/브라우저 **고유의** 동작 원리와 제약, 다른 생태계와 다르게 동작하는 지점, 실무에서 갈리는 설계 판단과 그 트레이드오프.
- **적극 활용하는 것**: 독자의 기존 경험에 빗댄 비교. 예: "이벤트 루프는 단일 스레드 기반의 논블로킹 I/O 모델로, Java의 스레드 풀 방식과 달리 …", "React의 재조정(reconciliation)은 가상 트리 diff 문제로, …"

### 깊이의 3층위 (모든 문서의 필수 기준)

경력 개발자는 표면적 사용법을 공식 문서로 스스로 익힌다. 이 저장소 문서의 가치는 공식 문서가 말해 주지 않는 층위에 있다. 모든 주요 주제는 다음 세 층위를 갖춰 서술한다.

1. **동작 모델** — 표면 API 아래에서 무슨 일이 일어나는가. 파서·엔진·런타임·네트워크 수준까지 내려가 메커니즘으로 설명한다. "그냥 그렇게 동작한다"는 서술을 남기지 않는다.
2. **설계 배경** — 왜 이렇게 설계되었는가. 어떤 대안이 있었고, 이 설계가 무엇을 얻기 위해 무엇을 포기했는가.
3. **경계 조건** — 이 기술/추상화/선택이 언제 무너지는가. 한계 상황, 성능이 급락하는 지점, 다른 선택으로 갈아타야 하는 신호.

"어떻게 쓰는가"에서 멈춘 섹션은 미완성으로 간주한다.

## 2. 문서 구조

모든 교육 문서는 아래 구조를 따릅니다. 각 섹션 제목은 그대로 사용합니다.

```markdown
# <챕터 번호>. <제목>

> 한 줄 요약: 이 문서를 읽고 나면 무엇을 판단/구현할 수 있는지 한 문장으로.

## 학습 목표
- 3~5개. "~을 안다"가 아니라 "~을 설명할 수 있다 / ~을 판단할 수 있다 / ~을 구현할 수 있다"로 서술.

## 배경: 왜 이것이 존재하는가
- 이 기술/개념이 해결하는 문제. 등장 전의 상태와 등장 배경, 경쟁했던 대안.
- 독자의 기존 경험(서버, 타 언어)과 연결되는 지점이 있으면 여기서 연결.

## 핵심 개념
- 본문. 동작 원리 중심으로 서술하며, 필요한 만큼 하위 섹션(###)으로 나눈다.
- 표면 API가 아니라 그 아래 계층(파서, 엔진, 런타임, 네트워크)에서 무슨 일이 일어나는지까지 내려간다.
- 모든 개념은 실행 가능한 코드 예제와 함께 설명한다.

## 실무 관점
- 실제 프로젝트에서 이 개념이 문제가 되는 상황, 흔한 오해와 안티패턴.
- 선택지가 갈리는 지점의 **트레이드오프 비교를 반드시 포함**한다 (표 활용 권장). 각 선택이 무너지는 경계 조건을 함께 명시한다.

## 더 깊이
- 스펙/엔진 내부 구현 수준의 심화 내용. 본문 흐름을 끊지 않도록 분리 배치.
- 심화 커리큘럼의 정체성에 해당하는 섹션이므로 원칙적으로 포함한다. 본문에서 이미 스펙/내부 수준까지 소화한 경우에만 생략할 수 있다.

## 정리
- 핵심을 5개 내외의 불릿으로. 새 정보 금지, 본문의 압축만.

## 확인 문제
- 2~4개. 단순 암기 확인이 아니라 "이 상황에서 어떤 일이 일어나는가 / 왜 이 코드는 의도대로 동작하지 않는가 / 이 설계에서 무엇을 선택하겠는가"류의 사고 문제.
- 정답과 해설은 <details> 태그로 접어서 제공.

## 참고 자료
- 1차 자료(스펙, 공식 문서) 우선. 링크마다 무엇을 얻을 수 있는지 한 줄 설명.
```

## 3. 서술 스타일

### 언어와 문체

- 모든 문서는 **한국어**로 작성한다. 문체는 **평서형(~한다/~이다)** 로 통일한다. 존댓말을 쓰지 않는다.
- 기술 용어는 첫 등장 시 원어를 병기한다. 예: 클로저(closure), 재조정(reconciliation). 이후에는 한국어 용어만 사용하되, 한국어 번역이 어색하거나 통용되지 않는 용어(hook, props, race condition 등)는 원어를 그대로 쓴다.
- 동료 시니어에게 설명하듯 쓴다. 독자를 가르치려 들지 않고("쉽죠?", "꼭 기억하세요!" 금지), 근거와 함께 담백하게 서술한다.
- 마케팅성 수식어("강력한", "혁신적인", "마법 같은")를 쓰지 않는다. 기술의 장점은 구체적 근거로 보인다.

### 설명 방식

- **원리 → 사용법** 순서로 설명한다. API 나열이 아니라, 동작 모델을 먼저 세우고 API는 그 모델의 표현으로 소개한다.
- **스펙의 용어와 모델을 기준으로 설명한다** (WHATWG HTML, ECMA-262, CSSWG 명세, Fetch Standard 등). 통념과 스펙이 다르면 스펙을 따르고 통념이 왜 생겼는지 짚는다.
- 엔진/브라우저 구현별로 다르게 동작하는 지점(V8/SpiderMonkey/JSC, Blink/WebKit/Gecko)은 그 사실 자체를 명시한다. 단정할 수 없는 것은 단정하지 않는다.
- **성능에 관한 주장은 확인 방법을 함께 제시한다.** "느리다"로 끝내지 않고, DevTools의 어느 패널에서 어떻게 관찰/측정하는지 또는 어떤 벤치마크 근거인지를 붙인다.
- 역사적 맥락은 현재의 설계를 이해하는 데 필요한 만큼만 다룬다. 레거시 기법(예: `var`, 콜백 지옥)은 "왜 대체되었는가"의 관점으로만 언급한다.
- 비유는 독자의 개발 경험 안에서 찾는다. 일상 사물 비유(레스토랑, 우체국 등)보다 타 기술 스택과의 비교가 효과적이다.

### 코드 예제

- 모든 예제는 **복사해서 바로 실행 가능한 최소 단위**로 작성한다. 생략이 필요하면 `// ...`로 명시한다.
- 최신 표준 문법 기준으로 작성한다 (ES2020+, 함수형 컴포넌트, `const` 기본).
- 나쁜 예를 보여줄 때는 반드시 좋은 예와 짝을 짓고, 주석으로 `// ❌`, `// ✅`를 표기한다.
- 예제 코드의 주석은 "무엇을 하는지"가 아니라 "왜 그렇게 하는지, 이 지점에서 무슨 일이 일어나는지"를 설명한다.
- 내부 동작을 설명할 때는 가능하면 **그 동작을 직접 관찰하는 예제**(실행 순서 출력, DevTools 확인 절차 등)를 함께 제공한다.
- 변수명/식별자는 영어, 주석은 한국어로 작성한다.
- 실행 결과가 중요한 예제는 출력을 주석(`// 출력: ...`)으로 함께 보여준다.

### 분량

- 문서 1개는 **45분~1시간 30분 안에 읽고 예제를 따라 할 수 있는 분량**을 유지한다 (대략 본문 2,000~4,000 단어). 깊이를 위해 분량이 필요하면 늘리는 쪽을 택하되, 상한을 넘으면 문서를 쪼갠다.
- 문서를 쪼갤 때는 ROADMAP.md의 문서 목록 갱신을 함께 제안한다.
- 분량 압박이 있을 때 줄이는 대상은 깊이가 아니라 **주제의 폭**이다. 얕게 여러 주제를 다루느니 한 주제를 3층위(동작 모델·설계 배경·경계 조건)로 다룬다.

## 4. 정확성 원칙

- 확신이 없는 스펙/버전/수치는 추측으로 쓰지 말고 공식 문서를 확인한 뒤 작성한다.
- 내부 동작 서술의 1차 자료는 스펙(WHATWG HTML, ECMA-262, CSSWG, Fetch, HTTP RFC)과 엔진/프레임워크의 공식 문서·소스다. 블로그 글은 검증용 참고로만 쓴다.
- 특정 라이브러리를 다룰 때는 문서 안에 **기준 버전을 명시**한다. 예: "이 문서는 React 19 기준이다."
- 브라우저 호환성이 갈리는 기능은 지원 현황을 언급한다 (Baseline 상태 참고).
- 구현 세부 사항(예: V8의 히든 클래스)과 표준이 보장하는 동작을 구분해서 서술한다. 구현 세부에 의존한 코드를 권하지 않는다.

## 5. 작업 워크플로

문서를 새로 작성하거나 수정할 때:

1. **ROADMAP.md의 해당 Phase 문서 목록을 확인**하고, 문서 번호·파일명·주제 범위를 그대로 따른다. 범위 변경이 필요하면 ROADMAP.md 수정을 먼저 제안한다.
2. 문서는 `docs/phase-N/NN-slug.md` 경로에 배치한다.
3. 같은 Phase의 앞선 문서에서 이미 다룬 내용은 반복하지 않고 상대 링크로 연결한다. 예: `[비동기 처리](./06-promises-and-async.md)`. 다른 Phase에서 심화할 주제는 "여기서는 ~까지만 다루고, 심화는 N-M에서 다룬다"고 위임 지점을 명시한다.
4. 문서 작성이 완료되면 **ROADMAP.md 5절의 진행 현황 표를 갱신**한다.
5. 커밋 메시지는 한국어로, `docs: Phase 2-6 비동기 JavaScript 문서 작성` 형식을 따른다.

## 6. 문서 품질 자가 점검

작성을 마치면 다음을 확인한다.

- [ ] 5년차 개발자가 이미 아는 내용을 설명하는 문단이 없는가?
- [ ] "왜"에 대한 답 없이 "어떻게"만 나열한 섹션이 없는가?
- [ ] 주요 주제가 3층위(동작 모델·설계 배경·경계 조건)를 갖췄는가? "그냥 그렇다"로 남은 메커니즘이 없는가?
- [ ] 선택지가 갈리는 지점마다 대안과 트레이드오프가 제시되었는가?
- [ ] 성능/비용 관련 주장에 관찰·측정 방법이 붙어 있는가?
- [ ] 모든 코드 예제가 실제로 실행 가능한가? (의심되면 실행해서 검증한다)
- [ ] 확인 문제가 본문을 다시 찾아 읽게 만드는 사고형 문제인가?
- [ ] 2장의 문서 구조와 섹션 제목을 그대로 따랐는가?
- [ ] ROADMAP.md 진행 현황 표를 갱신했는가?
