# Phase 10 실습 과제 — 포트폴리오 프로젝트와 기술 검증 기록

Phase 10 문서 학습과 병행하는 최종 실습 과제다. 이 Phase의 목표는 앱 하나를 "돌아가게" 만드는 것이 아니라, 문제 정의부터 배포, ADR, 리뷰, 검증, 포트폴리오 서사까지 하나의 증거 체인으로 남기는 것이다.

최종 산출물은 다음 다섯 묶음이다.

- 배포 가능한 포트폴리오 프로젝트
- 요구사항 문서와 ADR 묶음
- 코드 리뷰·리팩터링·품질 검증 기록
- README 또는 사례 연구
- 이력서 bullet과 면접 답변 노트

## 1주차 — 프로젝트 정의와 ADR

[10-1 프로젝트 가이드](../../docs/phase-10/01-project-guide.md)와 병행한다.

### 과제 A. 주제와 판단 지점 정의

- [ ] 프로젝트 주제를 하나 선택했다.
- [ ] 단순 CRUD에 머무르지 않도록 데이터 신선도, 인증·권한, 검색·필터링, 대용량 목록, 오프라인 저장, 성능, 접근성, 보안 중 최소 3개 이상의 판단 지점을 포함했다.
- [ ] 핵심 사용자와 핵심 작업을 3개 이내로 정의했다.
- [ ] 범위 제외 항목을 명시했다.

### 과제 B. 요구사항 문서 작성

`docs/requirements.md` 또는 같은 목적의 문서를 만든다.

- [ ] 문제와 사용자 상황을 설명했다.
- [ ] 핵심 사용자 플로우를 단계로 작성했다.
- [ ] 기능 요구사항을 검증 가능한 동작으로 썼다.
- [ ] 비기능 요구사항에 성능, 접근성, 보안, 운영, 브라우저 지원 범위를 포함했다.
- [ ] 각 비기능 요구사항에 측정 조건 또는 검증 방법을 붙였다.

### 과제 C. ADR 작성

`docs/adr/` 아래에 최소 5개의 ADR을 작성한다.

추천 주제:

- 렌더링 전략
- 상태·데이터 계층
- URL 상태와 서버 상태 분리
- 스타일링 전략
- 테스트 범위
- 배포·캐시 전략
- 인증·토큰 저장 전략
- 브라우저 저장소 선택

각 ADR은 다음 항목을 포함한다.

- [ ] 상태: Proposed / Accepted / Superseded
- [ ] 맥락
- [ ] 선택지
- [ ] 결정
- [ ] 결과와 비용
- [ ] 검증 방법
- [ ] 재검토 조건

## 2주차 — 구현, 리뷰, 품질 검증

[10-2 코드 품질과 리뷰](../../docs/phase-10/02-code-quality-and-review.md)와 병행한다.

### 과제 D. 핵심 수직 절편 구현

- [ ] 화면, 데이터 흐름, 오류 상태, 배포 경로가 얇게라도 끝까지 연결된 수직 절편을 구현했다.
- [ ] loading, empty, failure, permission 또는 validation 오류 중 프로젝트에 맞는 경계 상태를 처리했다.
- [ ] 키보드만으로 핵심 플로우를 완주했다.
- [ ] preview 또는 production 배포 URL에서 동작을 확인했다.

### 과제 E. PR 또는 셀프 리뷰 기록

팀 프로젝트는 실제 PR을, 개인 프로젝트는 PR 본문 형식의 셀프 리뷰를 남긴다.

각 기록에는 다음을 포함한다.

- [ ] 문제와 변경 의도
- [ ] 관련 요구사항 또는 ADR 링크
- [ ] 검증 명령
- [ ] 스크린샷, trace, Network waterfall, Profiler 결과 중 관련 증거
- [ ] 접근성·성능·보안 영향
- [ ] 남은 리스크

최소 3개 이상의 리뷰 기록을 남긴다.

### 과제 F. 품질 gate 구성

프로젝트 규모에 맞게 다음 중 필요한 gate를 구성한다.

- [ ] typecheck
- [ ] lint
- [ ] unit 또는 component test
- [ ] e2e smoke test 또는 수동 검증 절차
- [ ] build
- [ ] 접근성 smoke check
- [ ] bundle 또는 performance budget
- [ ] secret 또는 dependency 보안 점검

gate가 느리거나 flaky하면 어떤 검사를 PR마다 돌리고 어떤 검사를 nightly/main 기준으로 돌릴지 기록한다.

### 과제 G. 리팩터링 기록

최소 1개의 리팩터링 기록을 남긴다.

- [ ] 변경 전 문제
- [ ] 보존해야 할 public behavior
- [ ] 변경 단위
- [ ] 검증 결과
- [ ] 남은 트레이드오프

## 3주차 — 포트폴리오, 이력서, 면접 답변

[10-3 포트폴리오와 이력서](../../docs/phase-10/03-portfolio-and-resume.md), [10-4 면접 준비](../../docs/phase-10/04-interview-prep.md)와 병행한다.

### 과제 H. README 또는 사례 연구 작성

README 또는 별도 case study에 다음을 포함한다.

- [ ] 문제와 목표
- [ ] 핵심 사용자 플로우
- [ ] 기술 선택 요약과 ADR 링크
- [ ] 아키텍처 또는 상태 경계 설명
- [ ] 로컬 실행 방법
- [ ] 환경 변수와 demo data 설명
- [ ] 테스트·빌드·검증 명령
- [ ] 성능·접근성·보안 점검 결과
- [ ] 알려진 한계와 실제 서비스 전 보강 항목

### 과제 I. 이력서 bullet 작성

프로젝트 기반 bullet을 3~5개 작성한다.

각 bullet은 다음 구조를 따른다.

```text
상황 또는 제약
  → 행동 또는 기술 선택
  → 결과 또는 검증 증거
```

정량 지표가 없으면 변경 전후의 관찰 가능한 품질 변화나 운영 리스크 감소를 쓴다.

### 과제 J. 면접 답변 노트 작성

- [ ] 프로젝트 기반 질문 10개 이상을 작성했다.
- [ ] Phase 0~9 원리 질문 20개 이상을 작성했다.
- [ ] 프론트엔드 시스템 설계형 질문 2개 이상을 작성했다.
- [ ] 각 답변에 동작 모델, 트레이드오프, 경계 조건, 검증 방법 중 빠진 층이 없는지 점검했다.
- [ ] ADR 하나를 골라 반대 선택지를 주장하고, 어떤 조건에서 결정을 바꿀지 답했다.

## 산출물 형식 예시

```text
project/
  README.md
  docs/
    requirements.md
    adr/
      001-rendering-strategy.md
      002-data-state.md
      003-testing-strategy.md
      004-deployment-cache.md
      005-security-boundary.md
    reviews/
      001-search-vertical-slice.md
      002-error-states.md
      003-performance-pass.md
    measurements/
      2026-07-07-lcp-baseline.md
      2026-07-08-accessibility-keyboard.md
      2026-07-09-security-input-path.md
    interview-notes.md
```

## 완성 기준

- [ ] 배포 가능한 앱이 있다.
- [ ] README만 보고 로컬 실행, 테스트, 빌드를 재현할 수 있다.
- [ ] 요구사항 문서에 사용자 시나리오, 기능/비기능 요구사항, 제외 범위가 있다.
- [ ] 주요 기술 선택 ADR 5개 이상이 있다.
- [ ] 수직 절편 구현과 배포 기록이 있다.
- [ ] typecheck/lint/test/build 중 프로젝트에 필요한 품질 gate가 있다.
- [ ] 주요 사용자 플로우의 loading, empty, failure, 권한 또는 입력 오류 상태를 처리했다.
- [ ] 성능 측정 결과와 해석 한계가 있다.
- [ ] 접근성 점검 결과가 있다.
- [ ] 보안 점검 결과가 있다.
- [ ] 코드 리뷰 또는 셀프 리뷰 기록 3개 이상이 있다.
- [ ] 리팩터링 전후의 문제·조치·검증 결과 기록 1개 이상이 있다.
- [ ] README 또는 case study가 문제, 제약, 선택, 측정, 한계를 설명한다.
- [ ] 이력서 bullet 3~5개가 작성되어 있다.
- [ ] 프로젝트 기반 면접 질문 10개 이상과 Phase별 원리 질문 20개 이상이 작성되어 있다.

## 진행 팁

- 프로젝트를 크게 만들기보다 판단 지점이 드러나는 작은 수직 절편을 먼저 끝낸다.
- ADR은 결정 직후 작성한다. 나중에 쓰면 실제 맥락이 사라진다.
- 성능·접근성·보안 점검은 성공한 결과뿐 아니라 실패한 기준선도 남긴다.
- 포트폴리오 문장은 "무엇을 썼다"보다 "어떤 제약에서 무엇을 포기했는가"를 먼저 쓴다.
- 면접 답변은 프로젝트 기록에서 시작한다. 추상 지식만 외우면 꼬리 질문에서 쉽게 끊긴다.
