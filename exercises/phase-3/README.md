# Phase 3 실습 과제 — 바닐라 JS 제작과 계측

Phase 3 문서 학습과 병행하는 3단계 실습 과제다. 각 과제는 [학습 기획](../../plan/phase3.md)의 주차 배분과 연동되며, 완성 기준(Definition of Done)을 체크리스트로 명시한다.

이 Phase의 실습은 Phase 2(관찰)와 달리 **제작 + 계측**이다. 프레임워크 없이 앱을 만들고, 만든 앱을 DevTools로 계측해 "돌아간다"에서 멈추지 않는다. 1~3주차는 각 문서의 예제를 직접 실행·변형하는 미니 실험으로 언어 감각을 세우고, 4주차부터 과제에 들어간다.

공통 제약:

- 프레임워크·번들러·UI 라이브러리를 쓰지 않는다. HTML + ESM(`<script type="module">`) + 브라우저 내장 API만 사용한다. 로컬 서버는 정적 서버(`npx serve`, `python -m http.server` 등) 하나면 된다.
- 코드는 최신 표준 문법(ES2020+, `const` 기본)으로 작성한다.
- 각 완성 기준의 "확인했다"는 DevTools 캡처 또는 실행 결과를 근거로 남기는 것을 뜻한다.

## 과제 A — 바닐라 JS Todo 앱 (4주차, 3-7까지 학습 후 착수)

localStorage 저장을 포함한 Todo 앱을 프레임워크 없이 제작한다. 요구사항이 곧 학습 검증이 되도록 설계했다 — 각 항목은 해당 문서의 개념을 코드로 이행하는지 확인한다.

### 기능 요구사항

- 항목 추가 / 완료 토글 / 삭제 / 남은 항목 수 표시
- 새로고침 후 상태 유지 (localStorage — [3-10](../../docs/phase-3/10-memory-and-storage.md))

### 구조 요구사항

**① 상태와 DOM의 분리**

상태 객체를 단일 출처(single source of truth)로 두고, 렌더 함수가 상태를 읽어 DOM을 갱신하는 구조로 만든다. DOM을 상태 저장소로 쓰지 않는다(예: 완료 여부를 클래스 유무로만 판별하지 않기). Phase 5의 "UI = f(state)"를 맨손으로 먼저 겪는 것이 목적이다.

- [ ] 모든 변경이 `상태 변경 → render(state)` 한 경로로만 흐른다 (이벤트 핸들러가 DOM을 직접 조작하는 지점이 없다)
- [ ] 속성(attribute)과 프로퍼티(property)의 구분([3-7](../../docs/phase-3/07-dom-and-events.md))을 렌더 함수에서 의식적으로 적용했다 — 체크박스 상태는 `checked` 프로퍼티로 다룬다

**② 이벤트 위임**

- [ ] 목록 항목(토글/삭제)의 리스너를 개별 부착하지 않고 목록 컨테이너 1곳에 위임했다 ([3-7](../../docs/phase-3/07-dom-and-events.md))
- [ ] `event.target.closest()`로 액션 대상을 판별하고, 항목을 추가/삭제해도 리스너 재부착이 필요 없음을 확인했다

**③ 모듈 구조 (5주차, 3-9 학습 후 리팩터링)**

- [ ] 상태 / 렌더 / 저장 계층을 ESM 모듈 3개 이상으로 분리했다 ([3-9](../../docs/phase-3/09-modules.md))
- [ ] 모듈 간 의존이 단방향인지(저장→상태→렌더 같은 순환이 없는지) 확인했다

### 완성 기준 (Definition of Done)

- [ ] 새로고침 후 상태가 유지된다
- [ ] 항목 100개를 넣은 상태에서 입력·토글에 체감 지연이 없고, Performance 패널 녹화에서 해당 인터랙션에 long task(50ms 초과)가 없음을 확인했다 ([3-5](../../docs/phase-3/05-event-loop.md))
- [ ] 렌더 함수가 목록을 갱신할 때 DOM 읽기와 쓰기가 뒤섞여 강제 동기 레이아웃을 유발하는 지점이 없는지 점검했다 ([3-7](../../docs/phase-3/07-dom-and-events.md))

## 과제 B — 공개 API 검색/조회 앱 (5주차, 3-8 학습 후)

공개 API(도서/영화/GitHub 저장소 검색 등 인증 없이 쓸 수 있는 것) 하나를 골라 **검색 → 목록 → 상세** 흐름의 앱을 제작한다. 네트워크 코드가 "행복 경로에서만 도는" 상태를 벗어나는 것이 목적이다.

### 요구사항

**① 입력 디바운스**

- [ ] 검색 입력에 디바운스를 직접 구현해 적용했다 (라이브러리 금지 — [3-2](../../docs/phase-3/02-closures-and-functions.md)의 클로저 응용)
- [ ] Network 패널에서 타이핑 중 요청이 발사되지 않고 입력이 멎은 뒤 1건만 나가는 것을 확인했다

**② 이전 요청 취소**

- [ ] 새 검색어 요청 직전에 이전 요청을 `AbortController`로 취소한다 ([3-6](../../docs/phase-3/06-promises-and-async.md), [3-8](../../docs/phase-3/08-network-apis.md))
- [ ] 빠르게 타이핑할 때 Network 패널에 `(canceled)` 요청이 관찰되고, 느린 이전 응답이 새 결과를 덮어쓰는 응답 역전이 발생하지 않음을 확인했다 (Network 스로틀링으로 재현)
- [ ] 취소로 발생하는 `AbortError`를 에러 UI로 표시하지 않고 무시하도록 구분 처리했다

**③ 에러의 3계층 구분 처리**

- [ ] 네트워크 에러(fetch reject)와 HTTP 에러(`response.ok === false`)를 구분해 다른 UI 메시지를 보여준다 ([3-8](../../docs/phase-3/08-network-apis.md))
- [ ] DevTools를 오프라인으로 전환했을 때와, 존재하지 않는 리소스를 요청했을 때(404) 서로 다른 에러 UI가 보이는 것을 확인했다

**④ 상태 표현**

- [ ] 로딩 / 성공 / 에러 / 빈 결과 4가지 상태가 UI에서 구분된다
- [ ] 상태 표현이 과제 A와 같은 "상태 객체 → 렌더" 구조를 따른다 (로딩 여부를 DOM에 직접 토글하지 않는다)

### 완성 기준 (Definition of Done)

- [ ] Network 패널 캡처: 디바운스 후 1건 발사 + 연속 타이핑 시 `(canceled)` 요청
- [ ] 오프라인 전환 캡처: 네트워크 에러 전용 UI
- [ ] 검색 결과가 없는 검색어에서 빈 결과 UI가 보인다

## 과제 C — 이벤트 루프/메모리 계측 분석 노트 (5주차, 3-10 학습 후)

과제 A 또는 B에서 만든 앱을 대상으로 계측한다. 산출물은 코드가 아니라 **원인 → 관찰 → 수정 → 재확인** 구조의 분석 노트다. Phase 7 성능 리포트의 축소 연습이다.

### 계측 항목

**① 태스크 분해 (3-5)**

- [ ] Performance 패널로 주요 인터랙션(항목 추가, 검색 등) 하나를 녹화해 이벤트 → JS 실행 → 렌더로 이어지는 경로를 태스크 단위로 분해해 기록했다
- [ ] long task(50ms 초과)의 유무를 확인하고, 있다면 어떤 코드가 차지하는지 flame chart로 특정했다

**② 누수 재현과 발견 (3-10)**

의도적으로 누수 버전을 만들고, 힙 스냅샷 비교로 그 누수를 "발견"하는 절차를 재현한다. 누수의 원인을 아는 상태에서 시작하지만, 노트는 원인을 모르는 사람이 발견해 가는 순서로 쓴다.

- [ ] 의도적 누수를 1개 이상 심었다 — 리스너 미해제 또는 detached DOM 참조 중 택 1 ([3-10](../../docs/phase-3/10-memory-and-storage.md)의 누수 패턴)
- [ ] 3-스냅샷 기법으로 반복 동작 전후의 힙 스냅샷을 비교해, 늘어나는 객체와 그 retainer 경로를 특정했다
- [ ] Detached 노드 검색(스냅샷 필터에 `Detached`)으로 분리된 DOM 트리를 확인했다 (detached 누수를 골랐을 경우)
- [ ] 수정(리스너 해제 또는 참조 제거) 후 같은 절차로 재계측해 증가가 멈춘 것을 확인했다

**③ localStorage 쓰기 비용 측정 (3-10)**

- [ ] `performance.mark`/`measure`로 localStorage 쓰기 소요 시간을 계측했다 — 저장 데이터 크기를 늘려가며(예: 100건 → 10,000건 직렬화) 소요 시간 변화를 기록했다
- [ ] 이 쓰기가 메인 스레드를 동기로 점유한다는 사실을 Performance 패널 태스크에서 확인하고, 쓰기 빈도/시점 설계(입력마다 vs 디바운스)에 대한 판단을 노트에 적었다

### 완성 기준 (Definition of Done)

- [ ] 세 항목 각각에 실제 캡처(Performance 트랙, 힙 스냅샷 비교 화면, measure 수치)가 근거로 첨부되어 있다
- [ ] 누수 항목이 "원인 → 관찰(발견 절차) → 수정 → 재확인" 4단 구조로 서술되어 있다
- [ ] 판단의 근거로 해당 문서(3-5, 3-10)의 동작 모델을 인용했다 (예: "이 태스크가 long task가 되는 이유는 …")

## 진행 팁

- 과제 A에서 "상태 → 렌더" 구조가 어색하게 느껴지는 지점(전체를 다시 그리면 입력 포커스가 날아간다 등)을 만나면 기록해 둔다 — Phase 5에서 React가 정확히 그 문제를 어떻게 푸는지 배울 때 가장 좋은 대조 재료다.
- 과제 B의 응답 역전은 정상 네트워크에서는 재현이 어렵다. Network 패널 스로틀링(Slow 3G)을 켜고 빠르게 타이핑해야 관찰된다.
- 힙 스냅샷은 찍기 직전에 GC가 강제 수행된다. "스냅샷에 남아 있다 = GC 후에도 도달 가능하다"로 읽으면 된다 ([3-10](../../docs/phase-3/10-memory-and-storage.md)).
- 계측 수치는 환경(기기, 확장 프로그램)에 따라 흔들린다. 절대값보다 수정 전후의 **변화**를 근거로 삼고, 시크릿 창에서 계측한다.
