# Phase 5 실습 과제 — React SPA와 리렌더 진단 리포트

Phase 5 문서 학습과 병행하는 실습 과제다. [학습 기획](../../plan/phase5.md)의 주차 배분과 연동되며, 완성 기준(Definition of Done)을 체크리스트로 명시한다.

이 Phase의 실습은 **제작 + 진단 리포트**다. SPA를 만드는 것으로 끝나지 않고, 렌더링 모델의 어휘(렌더/커밋, 전파, 신원)로 자기 앱의 리렌더를 계측·개선한 증거를 남긴다 — "만들 줄 안다"가 아니라 "왜 이렇게 동작하는지 진단할 수 있다"가 목표다.

공통 제약:

- React 19 + TypeScript(strict) + Vite. 예제·프로젝트 모두 `<StrictMode>`를 켠 상태를 기본으로 한다.
- [Phase 4의 타입 원칙](../phase-4/README.md)을 그대로 적용한다: 판별 유니언 상태, DOM/API 경계의 unknown + 가드, 근거 없는 단언 0개.
- 성능 관련 판단은 전부 React DevTools Profiler 계측을 근거로 한다. "느낌"으로 memo를 붙이지 않는다.

## 1~2주차 — 관찰 실험 (5-1 ~ 5-4와 병행)

문서 진도에 맞춰 최소 재현을 만들고, 각 결과를 예측 → 실행 관찰 → 한두 문장의 해설로 기록한다.

### 실험 A. 렌더와 재조정 (5-1, 5-2 학습 후)

- [ ] 컴포넌트에 렌더 로그를 심고, StrictMode의 이중 호출과 "부모 렌더 → 자식 렌더(props 무관)" 전파를 관찰했다
- [ ] 렌더 중 정의한 내부 컴포넌트가 부모 리렌더마다 입력 상태를 파괴하는 것을 재현하고, 모듈 최상위로 옮겨 해소했다
- [ ] index key 목록에서 맨 앞 삽입 시 입력값이 밀리는 것을 재현하고, 안정적 id key로 수정했다. `key={userId}`로 폼을 의도적으로 리셋하는 실험도 수행했다
- [ ] 상태 보유 컴포넌트가 children을 받는 구조로 바꿔, 자기 상태 갱신에도 children이 렌더되지 않는 것을 로그로 확인했다

### 실험 B. 상태·배칭·이펙트 (5-3, 5-4 학습 후)

- [ ] 한 핸들러에서 `set(count + 1)` 3연타와 `set(c => c + 1)` 3연타의 결과·렌더 횟수를 예측하고 검증했다
- [ ] 상태 객체를 변형(mutate) 후 같은 참조로 set하면 리렌더가 없는 것을 재현하고, 불변 갱신으로 수정했다
- [ ] 빈 의존성 배열 + 인터벌의 stale closure를 재현하고, 함수형 갱신으로 의존성 자체를 제거했다
- [ ] 지연이 다른 가짜 fetch 두 개로 race condition(늦은 응답이 이김)을 재현하고, AbortController(또는 ignore 플래그) 클린업으로 수정했다

## 3~5주차 — SPA 제작 + 진단 리포트

상품 목록/상세/장바구니 SPA를 만든다(공개 더미 API 또는 로컬 JSON 서버 사용 가능).

### 요구사항

**① 라우팅 ([5-7](../../docs/phase-5/07-routing-and-code-splitting.md))**

- [ ] React Router 7로 목록(`/products`) / 상세(`/products/:id`) / 장바구니(`/cart`)를 중첩 라우트 + 공통 레이아웃(Outlet)으로 구성했다
- [ ] 필터·정렬·페이지는 URL 쿼리 파라미터가 원본이다 — 새로고침·URL 공유로 같은 화면이 재현된다. 필터 변경은 replace로 처리해 뒤로가기가 페이지 단위로 동작한다
- [ ] 라우트 단위 lazy 분할 + 콘텐츠 영역 Suspense 경계를 적용하고, 빌드 산출물에서 청크 분리를 확인했다

**② 서버 상태 ([5-8](../../docs/phase-5/08-server-state.md))**

- [ ] 목록·상세를 TanStack Query 5로 관리한다. 쿼리 키에 queryFn이 읽는 입력(필터, 페이지, id)이 전부 들어 있다
- [ ] staleTime을 데이터 성격에 따라 다르게 선언하고 근거를 주석으로 남겼다
- [ ] 최초 로딩(pending)과 백그라운드 재검증(success + isFetching)의 UI를 구분했다 — 재검증에서 화면이 스피너로 덮이지 않는다
- [ ] 서버 데이터를 useState/스토어로 복사하는 코드가 없다 (이중 원본 금지)

**③ 클라이언트 상태 ([5-6](../../docs/phase-5/06-state-architecture.md))**

- [ ] 장바구니를 Context 또는 Zustand로 구현하고, **선택 근거**(전파 범위·갱신 빈도·부분 구독 요구)를 리포트에 기록했다
- [ ] Context 선택 시: Provider는 children을 받는 분리 컴포넌트이고 value 참조가 고정되어 있다 / Zustand 선택 시: 헤더 배지는 개수만 selector로 구독한다
- [ ] 지역으로 충분한 상태(캐러셀 슬라이드, 아코디언 등)가 전역에 올라가 있지 않다

**④ 타입 원칙 (Phase 4 연동)**

- [ ] `tsc --noEmit`(strict) 통과, 명시적 any 0개
- [ ] 뷰 상태·장바구니 액션이 판별 유니언으로 모델링되어 있고, API 응답 경계는 unknown + 가드(또는 스키마 검증)로 처리했다

### Profiler 리포트 (산출물)

리렌더 진단 리포트를 `report.md`로 작성한다. 개선 전 코드는 브랜치나 태그로 보존해 전/후를 재현 가능하게 한다.

- [ ] **의도적 초기 구현**: 검색어·필터 상태를 페이지 최상위에 두는 등 넓은 전파를 가진 버전으로 시작했다
- [ ] **계측**: Profiler("왜 렌더되었는가" 기록 켬)로 대표 상호작용 3개(검색 타이핑, 장바구니 담기, 페이지 이동)를 녹화하고, 각 커밋에서 무엇이 왜 렌더되는지 flame chart 스크린샷과 함께 기록했다
- [ ] **개선**: 구조적 해법(상태 내리기, children 패턴)을 먼저 적용하고, 그것으로 안 끊기는 지점에만 memo + 참조 안정화를 적용했다 — 각 조치의 근거(어느 인수를 줄이는가: 빈도/범위/무게)를 기록했다
- [ ] **재계측**: 같은 상호작용을 다시 녹화해 전/후 커밋 시간·렌더 범위를 비교했다. 효과 없는 조치는 되돌리고 그 사실도 기록했다

### 완성 기준 (Definition of Done)

- [ ] 세 페이지가 URL 계약(새로고침·공유·뒤로가기)을 지키며 동작한다
- [ ] `pnpm build` 성공 + 라우트 청크 분리 확인, `tsc --noEmit` 통과
- [ ] StrictMode에서 콘솔 경고(이펙트 중복 부작용, key 경고) 0개
- [ ] Profiler 리포트 완성 (전/후 계측 + 원인-조치-효과 서술)

## 진행 팁

- 상태를 만들 때마다 5-6의 판정 순서를 자문한다: 서버가 원본인가 → URL이 원본인가 → 지역으로 충분한가 → 그제야 전역. 이 습관이 리포트의 "장바구니 선택 근거"를 저절로 채워 준다.
- Profiler에서 "전부 렌더되는" 화면을 보고 놀라지 말 것 — 리렌더 자체는 문제가 아니다(5-5). 커밋 시간이 유의미한 상호작용부터 개선 대상으로 삼는다.
- 개선 전 버전을 지우지 말 것. 전/후 비교가 이 과제의 산출물이며, "개선했더니 수치가 안 변했다"도 유효한 기록이다(그 조치를 되돌리는 근거가 된다).
- 새 메시지/데이터가 잘 안 만들어질 때는 브라우저 DevTools의 네트워크 스로틀링으로 재검증·race 상황을 느리게 재현할 수 있다.
