# Phase 4 실습 과제 — TypeScript 마이그레이션

Phase 4 문서 학습과 병행하는 실습 과제다. [학습 기획](../../plan/phase4.md)의 주차 배분과 연동되며, 완성 기준(Definition of Done)을 체크리스트로 명시한다.

이 Phase의 실습은 새 코드 제작이 아니라 **마이그레이션 + 타입 설계 기록**이다. 이미 동작하는 코드([Phase 3 과제](../phase-3/README.md)의 Todo 앱 또는 API 검색 앱)의 암묵적 계약을 타입으로 명시화하면서, 추론이 실패하는 지점과 단언이 필요한 지점을 판정 규칙([4-1](../../docs/phase-4/01-type-system-foundations.md))으로 설명하는 훈련이다.

공통 제약:

- TypeScript 5.9, `strict: true` 전체 활성화. 빌드 도구를 새로 도입하지 않는다 — `tsc`로 트랜스파일해 기존 정적 서버로 실행하거나, 검사는 `tsc --noEmit`으로 분리한다([4-5](../../docs/phase-4/05-compiler-and-config.md)의 검사/방출 분리를 도구 구성으로 체험하는 것이 목적이다).
- 목표는 단언 0개가 아니라 **근거 없는 단언 0개**다. 모든 `as`/`!`에 근거를 기록한다.

## 1주차 — 판정 실험 (4-1 ~ 4-3과 병행)

문서 진도에 맞춰 짧은 실험을 수행하고, 각 결과를 예측 → `tsc --noEmit` 검증 → 한두 문장의 해설로 기록한다.

### 실험 A. 할당 가능성 예측 (4-1 학습 후)

- [ ] 초과 프로퍼티가 있는 객체를 ① 리터럴 직접 전달 ② 변수 경유 ③ `as` 단언 3가지로 함수에 넘겨 각각의 판정 결과를 예측하고 검증했다
- [ ] `const`/`let`/`as const` 3가지 선언의 추론 타입을 hover로 비교하고 넓히기 규칙으로 설명했다
- [ ] 콜백 안에서 좁히기가 무효화되는 사례를 재현하고, const 스냅샷으로 해소했다

### 실험 B. 상태 모델링 (4-2 학습 후)

- [ ] boolean 플래그 2개 이상으로 표현된 상태(자신의 Phase 3 코드에서 찾는다)를 판별 유니언으로 재설계했다
- [ ] assertNever 철저성 검사를 걸고, 유니언에 variant를 하나 추가해 컴파일 에러가 나는 것을 확인한 뒤 되돌렸다

### 실험 C. 변성 구멍 재현 (4-3 학습 후)

- [ ] `Dog[]` → `Animal[]` 할당 후 push로 배열을 오염시키는 코드를 작성해, 컴파일 통과 + 런타임 TypeError를 재현했다
- [ ] 같은 코드에서 매개변수를 `readonly Animal[]`로 바꿔 push가 컴파일 에러가 되는 것을 확인했다

## 2주차 — 마이그레이션 과제 (4-1 ~ 4-3 학습 후 착수, 4-4 ~ 4-5 병행)

Phase 3 과제 A 또는 B 하나를 선택해 TypeScript로 전환한다.

### 요구사항

**① strict + any 0개**

- [ ] `strict: true`, `noImplicitAny` 포함 전체 활성화에서 `tsc --noEmit`이 통과한다
- [ ] 명시적 `any`가 0개다 — 암묵 any 에러가 났던 지점마다 "왜 추론이 실패했는가"를 기록했다

**② 상태의 판별 유니언화 ([4-2](../../docs/phase-4/02-type-design.md))**

- [ ] 로딩/성공/에러/빈 결과(또는 Todo의 필터 상태 등)를 판별 유니언으로 재설계했다
- [ ] 렌더 함수의 switch에 assertNever 철저성 검사를 걸었다

**③ DOM 경계의 타입 처리 ([4-1](../../docs/phase-4/01-type-system-foundations.md))**

- [ ] `querySelector` 반환의 null과 요소 타입을 `!`/`as`가 아니라 가드(null 검사, instanceof)로 처리했다
- [ ] 단언이 불가피했던 지점(있다면)은 "왜 검사기는 모르고 나는 아는가"를 기록했다

**④ API 응답 경계 ([4-1](../../docs/phase-4/01-type-system-foundations.md)의 타입 소거)**

- [ ] fetch 응답을 제네릭 단언(`as User[]`) 대신 `unknown`으로 받고, 직접 작성한 타입 가드로 검증해 내부로 흘렸다 (과제 B 선택 시)
- [ ] localStorage에서 읽은 값도 같은 원칙으로 처리했다 (과제 A 선택 시 — 저장 포맷이 바뀐 옛 데이터가 남아 있는 시나리오를 고려한다)

**⑤ 도구 구성 ([4-5](../../docs/phase-4/05-compiler-and-config.md))**

- [ ] `isolatedModules`(또는 `verbatimModuleSyntax`)를 켜고, 타입 re-export가 있다면 `export type`으로 정리했다
- [ ] 검사(`tsc --noEmit`)와 실행 산출물 생성이 분리된 구성임을 스스로 설명할 수 있다

### 타입 설계 문서 (산출물)

- [ ] 모든 타입 단언(`as`)·non-null 단언(`!`)에 대해 "왜 추론이 실패했는가 / 왜 이 단언이 안전한가"를 기록했다
- [ ] 마이그레이션 중 발견한 기존 코드의 실제 버그(타입이 드러낸 null 처리 누락 등)를 목록화했다 — 이 목록이 정적 타입의 투자 대비 효과 증거다
- [ ] 판별 유니언 재설계 전/후의 타입 정의를 나란히 싣고, 재설계가 제거한 무효 상태를 명시했다

### 완성 기준 (Definition of Done)

- [ ] `tsc --noEmit` 통과 (strict 전체 + isolatedModules)
- [ ] 명시적 any 0개, 근거 없는 단언 0개
- [ ] 앱의 동작이 마이그레이션 전과 동일하다 (Phase 3 과제의 완성 기준을 다시 통과)
- [ ] 타입 설계 문서 완성

## 진행 팁

- 마이그레이션 순서는 의존 그래프의 잎(유틸리티, 상태 정의)부터 뿌리(진입점)로. 상태 타입을 먼저 판별 유니언으로 세우면 나머지 파일의 에러가 "고쳐야 할 목록"으로 정리된다.
- 에러가 수십 개 쏟아질 때 `!`로 진압하고 싶은 순간이 학습 지점이다 — 에러 메시지의 마지막 줄(실제로 어긋난 원자적 판정)부터 읽는 습관을 들인다.
- `tsc --noEmit --watch`를 띄워 두면 검사가 저장마다 증분으로 돈다 — 에디터(tsserver)와 tsc가 같은 tsconfig를 보는지 먼저 확인한다.
- 타입이 드러낸 기존 버그를 만나면 반갑게 기록한다. "마이그레이션했더니 버그가 나왔다"가 아니라 "버그가 이미 있었고 이제 보인다"가 정확한 서술이다.
