# Phase 2 실습 과제 — HTTP 트래픽 분석

Phase 2 문서 학습과 병행하는 2단계 실습 과제다. 각 단계는 [학습 기획](../../plan/phase2.md)의 주차 배분과 연동되며, 완성 기준(Definition of Done)을 체크리스트로 명시한다.

이 Phase의 실습은 코드 작성이 아니라 **관찰과 계측**이다. 도구는 두 가지면 충분하다 — `curl -v`(요청/응답 원문)와 DevTools Network 패널(타이밍·캐시·프로토콜). 모든 결과물은 캡처한 실제 헤더를 근거로 작성한다.

## 과제 A — 프로토콜 미니 실험 (1주차, 2-1 ~ 2-3과 병행)

문서 진도에 맞춰 짧은 실험 3개를 수행하고, 각각 캡처 + 한두 문단의 해설로 기록을 남긴다.

### A-1. 메시지와 의미론 관찰 (2-1 학습 후)

임의의 실서비스에 `curl -v`로 요청을 보내 요청/응답 원문을 캡처한다.

- [ ] 요청·응답의 시작줄과 주요 헤더 각각에 "무엇을 선언/협상하는 것인지" 주석을 달았다
- [ ] `curl -vL`로 리다이렉트 체인을 추적해 상태 코드(301/302/307 등)와 메서드 보존 여부를 확인했다
- [ ] `-X POST`로 같은 리다이렉트를 따라가며 메서드가 어떻게 되는지 302와 307에서 비교했다 (httpbin.org 활용 가능)
- [ ] 같은 URL에 `Accept-Encoding`을 바꿔 보내 다른 표현(압축)이 협상되는 것과 `Vary` 헤더를 확인했다

### A-2. 캐시 경로 3종 재현 (2-2 학습 후)

같은 리소스가 서버에 닿지 않는 세 가지 경로를 모두 직접 만들어 본다.

- [ ] `curl -sI`로 정적 자산의 `Cache-Control`/`ETag`/`Age`를 캡처하고 신선도 수명과 남은 시간을 계산했다
- [ ] 받은 ETag를 `If-None-Match`로 되돌려 보내 **304** 응답을 재현했다
- [ ] DevTools Network에서 `(memory cache)` / `(disk cache)` / `304` 세 경우를 각각 만들어 스크린샷으로 구분했다 (탭 내 재방문, 새 탭, 일반 새로고침을 조합)
- [ ] 일반 새로고침과 강력 새로고침의 요청 헤더 차이(`max-age=0` vs `no-cache`)를 확인했다

### A-3. 쿠키 속성 감사 (2-3 학습 후)

실서비스 하나에 로그인해 세션 쿠키의 속성을 감사한다.

- [ ] Application 패널에서 세션 쿠키의 `Domain`(host-only 여부), `Secure`, `HttpOnly`, `SameSite`, 만료를 표로 정리했다
- [ ] 각 속성에 대해 "무엇을 막고 있는가 / 빠져 있다면 어떤 위험인가"를 한 줄씩 평가했다
- [ ] Network 패널로 어떤 요청에 쿠키가 실리고 어떤 요청(다른 서브도메인, 서드파티)에 안 실리는지 확인했다
- [ ] `curl -c`/`-b`로 Set-Cookie 저장 → 재전송 사이클을 재현했다

## 과제 B — 실서비스 트래픽 분석 리포트 (2주차, 2-4 ~ 2-5 학습 후)

실서비스 하나를 골라(본인이 자주 쓰는 서비스 권장, 규모가 있어 CDN·H2 이상을 쓰는 곳) HTTP 트래픽을 항목별로 분석하고, **평가 + 개선안**을 담은 리포트를 작성한다. 단순 나열이 아니라 "이 정책이 왜 이렇게 되어 있는가 / 무엇을 바꾸면 무엇이 좋아지는가"까지 쓰는 것이 목적이다.

### 분석 항목 (리포트 필수 목차)

**① 프로토콜 버전**

- [ ] Network 패널 Protocol 열로 리소스별 h3/h2/http/1.1 분포를 집계했다
- [ ] `Alt-Svc` 헤더 또는 DNS HTTPS 레코드로 H3 광고 여부를 확인했다
- [ ] 오리진(도메인) 수를 세고, 도메인 샤딩 흔적이나 통합 여지를 평가했다

**② 캐싱 정책**

- [ ] HTML / 해시 파일명 자산 / 이미지 / API 응답 유형별로 `Cache-Control`을 표로 정리했다
- [ ] 불변 자산 패턴(콘텐츠 해시 + 장기 max-age + immutable) 적용 여부를 평가했다
- [ ] HTML 진입점의 재검증 구조(ETag/304)를 확인했다
- [ ] `Age`·`x-cache`류 헤더로 CDN 캐시 적중 여부를 관찰했다

**③ 쿠키**

- [ ] 과제 A-3의 감사 방법으로 전체 쿠키를 평가했다 (보안 속성 누락, 불필요하게 넓은 Domain/Path, 총 크기)
- [ ] 쿠키가 정적 자산 요청에도 실리는지, 그 바이트 비용을 추정했다

**④ 압축·콘텐츠 협상**

- [ ] 텍스트 자산의 `Content-Encoding`(br/gzip/zstd)과 무압축 대비 크기 차이를 확인했다
- [ ] `Vary` 헤더의 구성이 캐시 적중률 관점에서 적절한지 평가했다

**⑤ TLS·보안 전송**

- [ ] Security 패널에서 TLS 버전과 인증서 체인을 확인했다
- [ ] `curl -w`로 DNS/TCP/TLS/TTFB 타이밍을 분해해 연결 수립 비용을 측정했다
- [ ] `Strict-Transport-Security` 유무와 max-age·preload 여부를 확인했다
- [ ] mixed content 경고가 콘솔에 없는지 확인했다

### 완성 기준 (Definition of Done)

- [ ] 5개 항목 각각에 실제 캡처(헤더 원문 또는 스크린샷)가 근거로 첨부되어 있다
- [ ] 항목마다 "현재 상태 → 평가(적절/과잉/누락과 그 이유) → 개선안"의 3단 구조로 서술했다
- [ ] 개선안 중 최소 2개는 기대 효과를 정량적으로 추정했다 (예: "TLS 1.3 전환 시 연결 수립 1 RTT 절감 = 실측 RTT ○○ms 기준 ○○ms")
- [ ] 판단의 근거로 해당 문서(2-1~2-5) 또는 RFC 조항을 인용했다

## 진행 팁

- 분석 중 "왜 이렇게 되어 있지?"를 만나면 그 항목이 리포트에서 가장 좋은 재료다 — 통념과 다른 설정에는 대개 이유(레거시, CDN 기본값, 의도적 트레이드오프)가 있다.
- DevTools Network 패널은 "Preserve log"를 켜고 관찰한다. 리다이렉트·재검증처럼 순간적으로 지나가는 응답이 분석의 핵심인 경우가 많다.
- `curl`과 브라우저의 결과가 다르면 그 차이 자체가 발견이다 — 브라우저만 보내는 헤더(쿠키, Accept 계열), HTTP/2 여부, 캐시 개입을 의심한다.
