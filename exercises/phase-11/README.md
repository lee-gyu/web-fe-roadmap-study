# Phase 11 실습 과제 — AI 에이전트로 프론트엔드 프로젝트를 안전하게 변경하고 검증하는 리포트

Phase 11 문서 학습과 병행하는 실습 과제다. 이 Phase의 목표는 에이전트에게 "돌아가는 코드"를 만들게 하는 것이 아니라, 저장소 준비 → 작은 작업 위임 → 하네스와 검증 설계 → 운영 규칙 작성까지 하나의 검증 가능한 증거 체인으로 남기는 것이다.

최종 산출물은 다음 일곱 묶음이다.

- Agent-ready 저장소 지시문
- 작업 명세 템플릿 5종 이상
- 에이전트 실행 리포트 (작은 작업 1개 이상 + 큰 작업 1개 이상)
- 하네스 설계표 (도구·권한·MCP·샌드박스·검증 계층)
- 팀 운영 플레이북 (권한·PR 증거 요건)
- 사람-에이전트 역할 매트릭스
- 미래 개발 시나리오와 개인 성장 로드맵

## 실습 환경 전제

- 터미널 기반 코딩 에이전트를 최소 1개 준비한다. Claude Code, Codex CLI, Pi 중 하나를 권장하며, 과제는 특정 제품에 종속되지 않는다.
- 에이전트 사용에는 구독 또는 API 비용이 든다. 비용·토큰 예산 자체를 [11-6](../../docs/phase-11/06-safety-permissions-and-governance.md)의 학습 소재로 삼는다.
- 실습 대상 저장소는 Phase 10 프로젝트를 기본값으로 한다. 없다면 테스트가 존재하고 빌드 가능하며 적당한 규모(수십~수백 파일)인 공개 프론트엔드 저장소를 대체로 쓴다.
- 작업 전 항상 깨끗한 git 작업 트리에서 시작하고, 에이전트 작업은 별도 브랜치에서 한다([11-6](../../docs/phase-11/06-safety-permissions-and-governance.md)).

## 1주차 — Agent-ready 저장소 준비

[11-1 AI 에이전트 개요](../../docs/phase-11/01-agent-mental-model.md), [11-2 작업 흐름](../../docs/phase-11/02-agentic-frontend-workflow.md), [11-3 컨텍스트와 지시문](../../docs/phase-11/03-context-and-instructions.md)과 병행한다.

### 과제 A. 저장소 지시문 작성

- [ ] Phase 10 프로젝트 또는 대체 저장소를 하나 선택했다.
- [ ] 저장소 루트에 프로젝트 지시문(`AGENTS.md` 또는 사용 도구가 읽는 파일)을 작성했다.
- [ ] 프로젝트 개요, 설치·실행·검증 명령, 디렉터리 구조와 소유권, 코딩 규칙, 변경 금지 영역, 브라우저 지원 범위를 포함했다.
- [ ] 도구를 두 개 이상 쓴다면 지시문 원본을 하나로 통일하거나 동기화 방법을 명시했다.
- [ ] 지시문의 각 문장이 판정 가능한지(도구로 검증되거나 구체 행동으로 서술됐는지) 점검했다.

### 과제 B. Agent-ready 문서 정리

- [ ] 상태 소유권 지도(서버 상태/클라이언트 상태/URL 상태)를 정리하거나 기존 문서를 링크했다.
- [ ] 디자인 토큰, 접근성 기준, API 계약(OpenAPI 또는 타입 정의) 문서의 위치를 지시문에서 링크했다.
- [ ] API 계약 문서를 에이전트 소비 기준(스키마-응답 드리프트, 필드 의미·단위, 오류 스키마, enum 목록)으로 점검하고 부족한 항목을 기록했다.
- [ ] "신입 테스트"를 수행했다: 에이전트에게 "이 프로젝트의 실행·검증 방법과 구조를 설명하라"고 시키고, 답의 오류를 지시문·문서 보강으로 반영했다.

### 과제 C. 작업 명세 템플릿 작성

- [ ] 버그 수정, 기능 추가, 테스트 보강, 리팩터링, 문서화 요청 각각에 대한 작업 명세 템플릿을 작성했다(총 5종 이상).
- [ ] 각 템플릿은 목표, 재현(해당 시), 범위와 경계, 성공 조건(검증 명령 포함), 멈출 조건을 갖춘다.
- [ ] UI 관련 템플릿에는 상태표(loading/empty/error 등), 접근성 검증, 뷰포트, 브라우저 실증 항목을 포함했다.

### 과제 D. 도구 지형 비교표 작성

- [ ] 제품형 에이전트(Codex, Claude Code, Antigravity 중 최소 2개)와 오픈소스 미니멀 하네스(Pi)를 표면·핵심 도구·권한 모델·확장 방식 기준으로 비교했다.
- [ ] 비교 시점과 확인한 공식 문서 출처를 기록했다.

## 2주차 — 작업 위임과 하네스·검증 설계

[11-4 하네스 엔지니어링](../../docs/phase-11/04-harness-engineering.md), [11-5 검증·관측·리뷰](../../docs/phase-11/05-verification-and-observability.md)와 병행한다.

### 과제 E. 작은 작업 위임

- [ ] 작은 버그 수정 또는 테스트 보강 작업을 과제 C의 템플릿으로 명세했다.
- [ ] 에이전트에게 먼저 탐색과 계획만 요청하고(읽기 전용), 계획을 검토한 뒤 편집을 허용했다.
- [ ] 작업 중 에이전트가 읽은 파일, 실행한 명령, 실패한 시도, 사람이 개입한 지점을 기록했다.
- [ ] typecheck, lint, test, build 중 프로젝트에 맞는 검증을 실행하고 결과(명령과 실제 출력)를 남겼다.
- [ ] PR 설명 또는 변경 리포트를 작성했다: 변경 의도, 검증 결과, 스크린샷 또는 로그, 남은 리스크 포함.

### 과제 F. 하네스 설계표 작성

- [ ] 사용 가능한 도구를 파일 시스템, shell, package manager, browser, DevTools, MCP, issue tracker, design tool, monitoring/log tool로 분류했다.
- [ ] 각 도구별 권한(읽기 전용 / 승인 후 쓰기 / 네트워크 허용 여부 / secret 접근 여부 / 파괴적 명령 차단 기준)을 표로 정리했다.
- [ ] 되돌림 가능성과 영향 범위 기준으로 자동 허용/승인 필요/차단 세 등급을 나눴다.

### 과제 G. 더 큰 작업 위임과 증거 패키지

- [ ] 더 큰 작업(기능 추가, 컴포넌트 리팩터링, Storybook·테스트 보강, 접근성 개선, 성능 개선 후보 분석 중 하나)을 수행했다.
- [ ] 증거 패키지를 구성했다: diff, 정적 검증 로그, 테스트 결과, 브라우저 스크린샷 또는 Playwright trace, 접근성/성능/네트워크 근거(해당 시), 실행 요약, 남은 리스크.
- [ ] 실패나 재작업이 있었다면 원인을 계층별(명세/컨텍스트/도구·권한/환경/하네스/모델)로 귀인했다.

## 2주차 마무리 — 안전·팀 운영

[11-6 안전·권한·팀 운영](../../docs/phase-11/06-safety-permissions-and-governance.md)과 병행한다.

### 과제 H. 안전·팀 운영 플레이북 작성

- [ ] 에이전트 사용 정책 초안을 작성했다: 허용 작업, 승인 필요 작업, 금지 작업, PR 필수 증거, secret 처리, 외부 도구 연결, 로그 보존 기간.
- [ ] prompt injection과 악성 저장소 지시문을 가정한 점검 목록을 만들었다.
- [ ] PR 템플릿 또는 리뷰 체크리스트에 AI 에이전트 사용 여부, 사용 도구, 검증 명령, 사람이 리뷰한 범위, 남은 리스크 항목을 추가했다.
- [ ] 반복 작업 자동화 후보를 3개 고르고, 각 후보의 가치·검증 기계화 가능성·피해 반경을 평가했다.

## 마무리 — 사람의 역할과 미래 시나리오

[11-7 사람의 역할과 개발의 미래](../../docs/phase-11/07-human-role-and-future-of-development.md)와 병행한다.

### 과제 I. 역할 매트릭스와 미래 시나리오

- [ ] 본인의 현재 프론트엔드 업무를 위임 가능/부분 자동화(사람 승인)/사람의 판단이 핵심 세 열로 분류한 역할 매트릭스를 작성했다.
- [ ] 팀 차원의 미래 개발 흐름 시나리오를 작성했다(예: 이슈 작성 → 에이전트 계획 → 사람 승인 → 구현 → 자동 검증 → 사람 리뷰 → 배포 승인). 시나리오마다 필요한 문서, 도구, 권한, 검증 기준, 실패 대응 절차를 붙였다.
- [ ] 3개월/6개월/12개월 단위의 개인 성장 로드맵을 작성했다: 프론트엔드 기본기, 검증 능력, 제품 판단, 보안·접근성 책임, 에이전트 하네스 운영 능력을 포함한다.

## 완성 기준 (Definition of Done)

- [ ] 프론트엔드 저장소 하나를 agent-ready 상태로 정리했다(지시문 + 링크된 문서).
- [ ] 작업 명세 템플릿 5종 이상을 작성했다.
- [ ] 제품형 에이전트와 오픈소스 하네스(Pi)의 성격·사용 범위 비교표를 작성했다.
- [ ] 작은 작업 1개 이상을 에이전트와 함께 수행하고 검증 결과를 기록했다.
- [ ] 기능 추가 또는 리팩터링 작업 1개 이상을 에이전트와 함께 수행하고 증거 패키지를 작성했다.
- [ ] typecheck/lint/test/build 중 프로젝트에 필요한 검증 명령 실행 기록이 있다.
- [ ] 브라우저 스크린샷, Playwright trace, Lighthouse, 접근성 점검 중 하나 이상으로 UI 결과를 검증했다.
- [ ] 도구·권한·MCP 사용 정책 표를 작성했다.
- [ ] 에이전트 실패 사례 또는 한계 사례 1개 이상을 원인별로 귀인했다.
- [ ] PR 템플릿 또는 리뷰 체크리스트에 AI 에이전트 사용 항목을 추가했다.
- [ ] 팀 운영 플레이북 초안을 작성했다.
- [ ] 사람-에이전트 역할 매트릭스를 작성했다.
- [ ] 1년 단위 미래 개발 시나리오와 개인 성장 로드맵을 작성했다.

## 진행 팁

- 지시문은 처음부터 완벽할 필요가 없다. 신입 테스트로 에이전트가 틀리는 지점을 찾아 반복 보강하는 편이 빠르다.
- 작업 명세의 성공 조건에 검증 명령이 없으면 위임하지 않는다 — 검증 명령이 안 써지는 작업은 아직 명확하지 않은 작업이다.
- 실패한 시도를 지우지 말고 기록한다. 실패 귀인 연습의 재료이자 하네스 개선의 근거가 된다.
- 증거 패키지의 "남은 리스크" 항목이 비어 있으면 의심한다. 대개 찾지 않은 것이지 없는 것이 아니다.
- 팀 운영 플레이북은 이상적 규칙보다 지금 저장소·팀 규모에 맞는 최소 규칙에서 시작한다.
