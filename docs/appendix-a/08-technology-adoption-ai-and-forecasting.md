# A-8. 기술 도입·AI·미래 예측

> 한 줄 요약: 신기술과 AI 도구를 과장과 냉소 사이에서 판단하고, 도입 결정을 검증 가능한 가설과 운영 비용으로 분해할 수 있다.

## 학습 목표

- Hype Cycle과 Amara's Law를 기술 기대와 실제 효과의 시간차로 설명할 수 있다.
- Bitter Lesson을 AI와 자동화 도구 도입에서 범용 방법, 계산 자원, 확장성의 관점으로 해석할 수 있다.
- Stochastic Parrot 논의를 언어 모델의 한계, 의미 이해 주장, 검증 책임의 관점에서 설명할 수 있다.
- 신기술 도입 결정을 문제 적합성, 실패 비용, 검증 가능성, 팀 역량, rollback 가능성으로 평가할 수 있다.

## 배경: 왜 이것이 존재하는가

프론트엔드 생태계는 빠르게 움직인다. 프레임워크, 번들러, 렌더링 전략, 타입 도구, 테스트 도구, 디자인 시스템, AI 에이전트가 계속 등장한다. 어떤 기술은 단기 과장 뒤에 사라지고, 어떤 기술은 처음에는 과소평가되다가 개발 방식 자체를 바꾼다. 문제는 현재 시점에서 둘을 구분하기 어렵다는 것이다.

AI 도구는 이 어려움을 더 크게 만든다. 자동완성, 코드 생성, 테스트 보강, 리팩터링, 브라우저 조작, PR 작성은 이미 실무에 들어왔다. 동시에 모델은 틀릴 수 있고, 근거 없는 자신감을 보이며, 보안 경계를 넘을 수 있고, 검증 비용을 사람에게 남긴다. 이 문서는 신기술을 믿거나 거부하는 문서가 아니라, 어떤 조건에서 제한적으로 도입하고 어떤 증거로 판단을 바꿀지 정하는 문서다.

이 문서에서 다루는 원문 항목명은 The Hype Cycle & Amara's Law, The Bitter Lesson, The Stochastic Parrot이다.

## 핵심 개념

### 단기 효과와 장기 효과는 다르게 보인다

The Hype Cycle은 신기술이 과도한 기대, 실망, 점진적 생산성의 단계를 거칠 수 있다는 모델이다. Amara's Law는 기술의 단기 효과를 과대평가하고 장기 효과를 과소평가하는 경향을 말한다. 두 관점은 함께 읽을 때 유용하다.

새 프레임워크나 AI 도구가 등장하면 초기 데모는 강렬하다. 데모는 대개 가장 잘 맞는 문제를 보여 주고, 운영 비용과 실패 모드는 숨긴다. 이후 실제 프로젝트에 적용하면 edge case, 팀 학습 비용, migration, 보안, 테스트, 생태계 미성숙이 드러난다. 이때 실망이 온다. 그러나 일부 기술은 이 실망 구간을 지나 도구, 문서, 패턴, 생태계가 안정되며 장기 효과를 만든다.

따라서 초기 반응은 두 극단을 피해야 한다. "모든 것이 바뀐다"도 성급하고, "데모일 뿐이다"도 성급하다. 더 나은 질문은 "어떤 제한된 문제에서 어떤 성공 기준으로 실험할 것인가"다.

### 범용 방법과 확장성은 장기적으로 강하다

The Bitter Lesson은 AI 역사에서 인간이 만든 도메인 특화 지식보다 계산 자원과 범용 방법을 활용하는 접근이 장기적으로 이기는 경향을 지적한다. 이 관찰은 AI 도구 도입에서 중요한 균형을 만든다. 지금 부족해 보이는 범용 모델이 계산, 데이터, 도구 사용 능력, 하네스 개선으로 빠르게 나아질 수 있다.

그러나 Bitter Lesson을 "규칙과 도메인 지식은 무의미하다"로 읽으면 위험하다. 실무 시스템에는 보안 정책, 제품 맥락, 코드베이스의 암묵적 계약, 조직 의사결정, 사용자 피해가 있다. 범용 모델이 강해질수록 오히려 검증 하네스, 권한 모델, 테스트, 관측 가능성, 인간의 문제 정의가 더 중요해진다.

프론트엔드에서 AI 에이전트가 작은 버그 수정과 테스트 보강을 잘하더라도, 제품 요구사항의 우선순위나 접근성 책임, 보안 예외 승인, 사용자 신뢰 판단을 자동으로 맡길 수는 없다. 모델의 능력과 시스템의 책임을 분리해야 한다.

### 언어 모델은 유창함과 진실성을 구분하지 않는다

The Stochastic Parrot 논의는 대규모 언어 모델이 통계적 패턴을 기반으로 유창한 언어를 생성하지만, 그 출력이 곧 의미 이해나 사실성을 보장하지 않는다는 비판을 담고 있다. 이 논의 자체도 여러 관점이 있으므로 단순 구호로 쓰지 않는 편이 좋다. 실무적으로 중요한 점은 모델 출력의 유창함이 검증 증거가 아니라는 것이다.

AI 에이전트가 "테스트를 통과했다"고 말해도 실제 명령 출력이 필요하다. "보안을 고려했다"고 말해도 threat model과 코드 경계가 필요하다. "최신 API를 사용했다"고 말해도 기준 버전과 공식 문서가 필요하다. 모델은 도움이 되지만, 검증 책임은 도구 바깥의 시스템과 사람에게 남는다.

### 도입 판단은 기술보다 실패 비용에서 시작한다

신기술 도입은 다음 질문으로 시작한다.

| 질문 | 의미 |
|---|---|
| 어떤 문제를 해결하는가 | 도구가 아니라 병목을 먼저 정의한다 |
| 실패하면 피해가 무엇인가 | 사용자 피해, 보안, 데이터 손실, 일정 지연을 본다 |
| 성공을 어떻게 측정하는가 | 속도, 결함률, 유지보수성, 비용을 관찰한다 |
| 팀이 운영할 수 있는가 | 학습 비용, 문서, debugging, hiring을 본다 |
| 되돌릴 수 있는가 | migration, lock-in, data format, public API를 본다 |
| 기존 선택지와 비교했는가 | 현재 도구 개선으로 해결 가능한지 확인한다 |

AI 도구 도입도 같다. "코드를 빨리 쓴다"는 목표는 너무 넓다. "테스트 없는 legacy util에 characterization test를 추가한다", "디자인 변경의 영향 파일을 탐색한다", "반복적인 API 타입 변환을 생성하되 타입체크와 리뷰를 통과해야 한다"처럼 제한된 문제와 검증 기준이 필요하다.

## 실무 관점

기술 도입은 단계적으로 진행한다.

| 단계 | 산출물 | 중단 조건 |
|---|---|---|
| 문제 정의 | 병목, 현재 비용, 성공 기준 | 해결할 문제가 도구보다 불명확하다 |
| 작은 실험 | 제한된 범위의 spike, 검증 결과 | 재현 가능한 개선이 없다 |
| 운영 검토 | 보안, 권한, 비용, observability | 실패 시 피해를 감당할 수 없다 |
| 점진 적용 | migration plan, rollback, 문서 | lock-in이 크고 되돌릴 경로가 없다 |
| 회고 | 예측과 실제 결과 비교 | 지표가 부작용을 숨긴다 |

AI 에이전트는 특히 권한과 검증 경계가 중요하다. 파일 수정 권한, 네트워크 접근, secret 접근, 패키지 설치, destructive command, 비용 예산을 분리한다. 결과는 타입체크, 테스트, lint, 빌드, 브라우저 확인, 코드 리뷰, 로그로 검증한다. 도구가 잘할수록 사람이 해야 할 일은 줄어드는 것이 아니라 더 상위의 문제 정의와 증거 판단으로 이동한다.

## 더 깊이

미래 예측은 맞히기보다 업데이트 가능해야 한다. 좋은 기술 판단 문서는 다음을 남긴다.

- 현재 믿고 있는 전제
- 이 기술이 해결한다고 보는 병목
- 선택하지 않은 대안과 이유
- 성공으로 볼 지표
- 실패나 중단으로 볼 신호
- 판단을 다시 검토할 날짜나 사건

이 구조는 Hype Cycle과 Amara's Law의 함정을 줄인다. 초기 기대에 휩쓸리지 않으면서도 장기 변화를 놓치지 않으려면, 예측을 기록하고 새로운 증거가 나오면 바꿔야 한다. "나는 처음부터 맞았다"보다 "어떤 신호에서 판단을 바꾸었는가"가 더 중요하다.

Bitter Lesson과 Stochastic Parrot도 충돌하는 렌즈로 함께 써야 한다. 전자는 범용 계산과 확장 가능한 방법의 장기 힘을 보게 하고, 후자는 유창한 출력과 이해·책임을 구분하게 한다. 하나만 붙잡으면 과장이나 냉소로 치우친다.

## 회고 질문

- 이 기술 도입은 어떤 실제 병목을 줄이는가?
- 초기 데모에서 보이지 않는 운영 비용과 실패 모드는 무엇인가?
- 성공 기준과 중단 기준은 측정 가능한가?
- AI 출력의 유창함을 어떤 검증 증거로 대체할 것인가?
- 어떤 신호가 나오면 지금의 판단을 바꿀 것인가?

## 정리

- 신기술은 단기적으로 과장되거나 과소평가될 수 있으므로 제한된 실험과 성공 기준이 필요하다.
- Bitter Lesson은 범용 방법과 확장성의 힘을 보게 하지만, 도메인 책임과 검증 하네스를 지우지 않는다.
- Stochastic Parrot 논의는 모델 출력의 유창함과 사실성, 책임을 구분하게 한다.
- 기술 도입은 문제 적합성, 실패 비용, 검증 가능성, 팀 역량, rollback 가능성으로 평가한다.
- 미래 예측은 맞히는 게임이 아니라 증거에 따라 업데이트하는 기록이다.

## 확인 문제

1. 팀이 새 AI 코딩 도구를 전면 도입하자고 한다. Hype Cycle 관점에서 어떤 절차가 필요한가?

<details>
<summary>정답과 해설</summary>

초기 데모만으로 전면 도입하지 말고 제한된 문제에서 실험해야 한다. 성공 기준, 실패 비용, 보안 권한, 검증 명령, rollback 경로를 정하고 실제 결과를 기록한다.

</details>

2. Bitter Lesson을 근거로 도메인 규칙과 테스트를 줄이자는 주장이 나왔다. 어떤 반론이 가능한가?

<details>
<summary>정답과 해설</summary>

Bitter Lesson은 범용 방법의 장기 힘을 말하지만, 실무 시스템의 보안, 제품 맥락, 사용자 피해, 검증 책임을 제거하지 않는다. 모델이 강해질수록 하네스와 테스트, 권한 모델이 더 중요해진다.

</details>

3. AI 에이전트가 "수정 완료, 테스트 통과"라고 보고했다. Stochastic Parrot 관점에서 무엇을 확인해야 하는가?

<details>
<summary>정답과 해설</summary>

보고 문장의 유창함은 증거가 아니다. 실제 command output, 변경 diff, 테스트 범위, 실패하지 않은 이유, 브라우저 확인, 남은 리스크를 확인해야 한다.

</details>
